### 実現可能な理由
1. **基盤が整っている**: Strapi + Next.js の構成は拡張性が高い
2. **コアロジック実装済み**: 予約フォーム・在庫チェックは完成度が高い
3. **スキーマ拡張が容易**: Strapiでフィールド追加は低コスト
4. **開発計画が存在**: 開発ノートでメール実装等が明確にToDo化されている

### 実現困難/時間がかかる機能
1. **LLM統合（AIゲートキーパー/AIアドバイザー）**: API連携・コスト管理・エラーハンドリングで2-4週間
2. **詳細モード（テーブル管理）**: スキーマ設計・UI・マイグレーションで3-6週間
3. **完全多言語対応**: テンプレート整備・翻訳で2-4週間

## 5. アーキテクチャ上の懸念点

### 5.1 Strapiの制約
- **カスタムロジック**: Strapiは基本的にCRUD主体。LLM統合には `controllers/` や `services/` でカスタム実装が必要
- **リアルタイム通知**: WebSocket等のリアルタイム機能は追加開発が必要

### 5.2 推奨される対応
| 懸念 | 対応策 |
|------|--------|
| メール送信 | Strapi Email Plugin + SendGrid or Resend |
| LLM統合 | カスタムコントローラーでOpenAI API呼び出し |
| 通知 | LINE Messaging API or Firebase Cloud Messaging |
| ステータス管理 | 予約スキーマに`status`を追加 + ワークフロー実装 |

## 7. 結論

この仕様書は**「現実的に実装可能」**ですが、**全機能を一度に実装するのは非現実的**です。

### 推奨アプローチ
1. **Phase 1（MVP強化）を最優先**で実装し、チラシに記載された機能を満たす
2. **LLM機能は後回し**で、まずは手動判定（備考欄があれば店主確認）で運用開始
3. **詳細モードは将来の有料アップグレード**として設計

### 工数見積もり
| Phase | 工数 | 必要スキル |
|-------|------|-----------|
| Phase 1 | 2-3週間 | Strapi、メール連携 |
| Phase 2 | 2-3週間 | LINE API、DB設計 |
| Phase 3 | 3-4週間 | LLM API、多言語 |
| **合計** | **7-10週間** | フルスタック |

---

> **TIP**: 現状の `checkAvailability` 実装は品質が高く、そのまま活用可能です。メール送信基盤の構築を最優先にすることで、最小限のコストで「仮受付→確定」フローを実現できます。
