- フロントエンド起動済み: `npm run dev` (frontend)
- バックエンド起動済み: `npm run develop` (backend)

### テスト手順

#### Step 1: ホーム画面への追加
1. iPhoneのSafariで管理画面URL（`https://<domain>/admin/dashboard`）を開く
2. 「共有」ボタン → 「ホーム画面に追加」を選択
3. **期待**: アイコンがホーム画面に追加される

#### Step 2: 業務開始ボタンのタップ
1. ホーム画面からアプリを起動
2. 「業務開始」ボタンをタップ
3. **期待**: AudioContextが初期化され、音声再生が有効化される
4. **確認方法**: デベロッパーツールで `[PWA] Audio initialized successfully` ログを確認

#### Step 3: 画面ロック中のモック予約トリガー
1. 業務開始後、iPhoneの画面をロック
2. 別端末で予約を作成（POST `/api/public/reservations`）

   ```bash
     -H "Content-Type: application/json" \
     -d '{
       "storeId": "<store-document-id>",
       "guestName": "テスト太郎",
       "email": "test@example.com",
       "phone": "090-1234-5678",
       "date": "2025-12-15",
       "time": "18:00",
       "guests": 2
     }'
   ```
3. **期待**: iPhoneで通知音が再生される

### 合格基準
- [ ] 通知音が画面ロック中でも再生される
- [ ] ビジュアルバッジ（新規予約カウント）が更新される

## 🧪 テストシナリオ 2: メール配信（AWS SES）

### 前提条件
- AWS SES サンドボックス環境で送信者/受信者アドレスが検証済み
- 環境変数設定済み:
  ```
  SMTP_HOST=email-smtp.ap-northeast-1.amazonaws.com
  SMTP_PORT=587
  SMTP_USER=<your-ses-smtp-user>
  SMTP_PASS=<your-ses-smtp-password>
  EMAIL_FROM=noreply@yourdomain.com
  ```

### 実装箇所
- [email.ts](file:///e:/15.app_sumaho_uketsuke/backend/src/services/email.ts)

### テスト手順

#### Step 1: 予約作成とPending メール確認
1. 予約を作成する
2. **期待**: `reservation_pending` テンプレートでメールが送信される
3. **確認**: 受信メールの変数置換を確認
   - `{{ guestName }}` → 予約者名
   - `{{ store.name }}` → 店舗名

#### Step 2: ダッシュボードで承認 → Confirmed メール確認
1. 管理画面で予約を開く
2. 「承認」ボタンをクリック
3. 返信メッセージを入力（例: `お待ちしております！`）
4. **期待**: `reservation_confirmed` テンプレートでメールが送信される

#### Step 3: 変数置換の確認
確認すべき変数:
| Handlebars変数 | 置換結果例 |
|---------------|-----------|
| `{{ reservation.guestName }}` | 山田 太郎 |
| `{{ reservation.dateFormatted }}` | 12月15日（日） |
| `{{ reservation.time }}` | 18:00 |
| `{{ reservation.guests }}` | 2名様 |
| `{{ reservation.ownerReply }}` | お待ちしております！ |
| `{{ store.name }}` | レストラン○○ |
| `{{ store.phoneNumber }}` | 03-1234-5678 |

#### Step 4: スマートフォンメールアプリでレイアウト確認
1. iPhoneまたはAndroidのメールアプリで受信メールを開く
2. **期待**: テーブルが崩れていない、レスポンシブ表示

### 合格基準
- [ ] メールが受信トレイに届く（サンドボックスモードでは検証済みアドレスのみ）
- [ ] 変数（`{{ guestName }}`, `{{ ownerReply }}`）が正しく置換されている

## 🧪 テストシナリオ 3: リスク管理ロジック（リードタイム）

### 前提条件
- 店舗の `minBookingLeadTime` が設定されている（デフォルト: 180分 = 3時間）

### 実装箇所
- [public-reservation.ts L60-84](file:///e:/15.app_sumaho_uketsuke/backend/src/api/public-reservation/controllers/public-reservation.ts#L60-L84)
- [store/schema.json L95-98](file:///e:/15.app_sumaho_uketsuke/backend/src/api/store/content-types/store/schema.json#L95-L98)

### テスト手順

#### Step 1: リードタイム制限内の予約を試行
1. 現在時刻から1時間後の予約を試行（minBookingLeadTime = 180分の場合）
   ```bash
   # 例: 現在 15:20 の場合、16:20 の予約を試行
   curl -X POST http://localhost:1337/api/public/reservations \
     -H "Content-Type: application/json" \
     -d '{
       "storeId": "<store-document-id>",
       "guestName": "直前予約さん",
       "email": "test@example.com",
       "date": "2025-12-12",
       "time": "16:20",
       "guests": 2
     }'
   ```
2. **期待**: エラーレスポンスが返される
   ```json
   {
     "error": {
       "message": "当日の直前予約はお電話にて承ります。",
       "details": {
         "code": "BOOKING_LEAD_TIME_VIOLATION",
         "minLeadTimeMinutes": 180
       }
     }
   }
   ```

#### Step 2: 店主エントリーでリードタイム無視を確認
1. `isOwnerEntry: true` で同じ時刻の予約を作成
   ```bash
   curl -X POST http://localhost:1337/api/public/reservations \
     -H "Content-Type: application/json" \
     -d '{
       "storeId": "<store-document-id>",
       "guestName": "電話受付分",
       "date": "2025-12-12",
       "time": "16:20",
       "guests": 2,
       "isOwnerEntry": true
     }'
   ```
2. **期待**: 予約が正常に作成される

### 合格基準
- [ ] リードタイム制限内の予約がエラーでブロックされる
- [ ] `isOwnerEntry: true` の場合はリードタイム制限がバイパスされる

## 🧪 テストシナリオ 4: 詳細モードロジック（満席）

### 前提条件
- 店舗の `maxCapacity` と `maxGroupsPerSlot` が設定されている
  - デフォルト: `maxCapacity: 20`, `maxGroupsPerSlot: 5`

### 実装箇所
- [store.ts (checkAvailability)](file:///e:/15.app_sumaho_uketsuke/backend/src/api/store/services/store.ts)

### テスト手順

#### Step 1: 容量上限まで予約を作成
1. 特定の時間枠（例: 2025-12-15 18:00）に予約を5組作成
   ```bash
   # 1組目
   curl -X POST http://localhost:1337/api/public/reservations \
     -H "Content-Type: application/json" \
     -d '{
       "storeId": "<store-document-id>",
       "guestName": "テスト1",
       "email": "test1@example.com",
       "date": "2025-12-15",
       "time": "18:00",
       "guests": 4
     }'
   
   # 2組目〜5組目も同様に...
   ```

#### Step 2: 6組目の予約を試行
1. 同じ時間枠に6組目の予約を試行
2. **期待**: エラーレスポンス（Sold Out）
   ```json
   {
     "available": false,
     "reason": "Maximum groups per slot (5) would be exceeded"
   }
   ```

### 合格基準
- [ ] maxGroupsPerSlot を超える予約が拒否される
- [ ] maxCapacity を超える人数の予約が拒否される

### ⚠️ 現状の制約
現在の実装は「簡易モード」（総人数/組数管理）のみ。
「詳細モード」（テーブル単位管理）は未実装です。
